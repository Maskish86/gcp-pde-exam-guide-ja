# Cloud Monitoring

Cloud Monitoring（旧Stackdriver）は、GCPサービスとアプリケーションからメトリクス/イベントを収集する。ダッシュボード、アラート、稼働時間チェック（uptime checks）により、データパイプラインやインフラの問題を素早く検知できる。

## ユースケース
- パイプラインのレイテンシ、失敗、データ鮮度を監視する。
- サービス健全性（ワーカーCPU/メモリ/ディスク、キューバックログ）を追跡する。
- 重要データセット/ジョブのSLA/SLO違反にアラートする。
- 運用可視化とインシデント対応のためのダッシュボードを作る。

## メンタルモデル
- メトリクスは、monitored resourceに紐づく時系列。
- アラートは定期的に条件を評価し、閾値超過で通知（page）する。
- ダッシュボードは単一システムに関連する複数メトリクスを可視化する。
- uptime checks は外部エンドポイントを検証し、失敗時にアラートを起動する。

## コア概念
- Metric：計測値（レイテンシ、エラー、バックログなど）。
- Monitored resource：計測対象（ジョブ、VM、サービスなど）。
- Alert policy：通知条件ルール。
- Notification channel：email/SMS/Slack/PagerDutyなど。
- Dashboard：関連メトリクスのビュー。

## よくあるパターン
- Pub/Subの最古未ACKメッセージ経過時間にアラート（バックログリスク）。
- Dataflowのwatermark遅延とスループットを追跡する。
- Dataprocのジョブ失敗とワーカー飽和を監視する。
- 行数やパイプラインラグにカスタムメトリクスを使う。

## 連携
- [[Processing/Dataflow|Dataflow]]: ジョブメトリクスとストリーミングラグ。
- [[Processing/Dataproc|Dataproc]]: クラスタとジョブの健全性。
- [[Storage/BigQuery|BigQuery]]: クエリとスロットメトリクス。
- [[Ingestion/PubSub|Pub/Sub]]: バックログサイズと配信経過時間。
- [[Cloud-Storage|Cloud Storage]]: リクエストとレイテンシメトリクス。

## セキュリティとアクセス制御
- ダッシュボード/アラートには最小権限の [[Security/IAM|IAM]] ロールを使う。
- アラート乱立を防ぐため、viewer（read）とeditor（write）を分離する。

## 運用と信頼性
- アラートポリシーはシステム/重大度で一貫した命名にする。
- 短時間スパイクのノイズを減らすため、multi-window alertingを使う。
- 迅速な切り分けのため、アラート説明にrunbookを記載する。

## よくある落とし穴
- 低シグナルのメトリクスにアラートし、ノイズを生む。
- ダッシュボード/アラートのプロジェクト単位スコープが曖昧。
- インフラ指標だけに依存し、パイプライン成果（outcome）を見ない。

## クイックチェックリスト
- 重要SLI（レイテンシ/鮮度/エラー率）を特定する。
- コアパイプラインと依存関係のダッシュボードを作る。
- 明確な閾値とrunbook付きでアラートポリシーを追加する。
- 通知チャネルとオンコールルーティングを確認する。
- アラートノイズをレビューし、月次でチューニングする。
